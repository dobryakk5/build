 **Чек-лист финальной валидации Е2**, чтобы добить именно рискованные места и понять, когда файл уже можно считать действительно выверенным.

## 1. Проверка структуры параграфов

Для каждого `paragraph` проверить:

* код существует и идёт в правильном порядке: `Е2-1-1` … `Е2-1-61`
* `title` не пустой
* `title` не содержит мусор вроде:

  * `§`
  * `Норма времени`
  * `табл.`
  * кусков шапки таблицы
* `unit` заполнен и выглядит как единица измерения, а не как обрывок фразы
* `application_notes` не содержат кусков таблицы
* `technical_characteristics` не содержат состава работ или звена

Цель:
убедиться, что карточка параграфа отделена от табличной части.

---

## 2. Проверка состава работ

Для каждого `paragraph_work_items`:

* каждый пункт — это именно действие, а не заголовок
* не должно быть строк вроде:

  * `ЭКСКАВАТОРЫ С МЕХАНИЧЕСКИМ ПРИВОДОМ`
  * `Норма времени и расценки`
  * `Таблица 1`
* пункты должны быть отдельными операциями, а не слепленным абзацем на полстраницы
* порядок пунктов должен быть естественным
* не должно быть дублей соседних строк

Особенно смотреть:

* длинные параграфы с несколькими вариантами работ
* места, где есть `а)`, `б)`, `в)` или отдельные режимы работы

---

## 3. Проверка состава звена

Для `paragraph_crew_items`:

* каждая запись должна описывать рабочего, а не примечание
* поля `profession`, `grade`, `count`, `raw` должны быть согласованы
* если автоматический разбор не смог выделить профессию или разряд, в `raw` всё равно должен остаться корректный исходный текст
* не должно быть попадания туда строк типа:

  * `Состав работы`
  * `Примечание`
  * `Норма времени`

Что проверить вручную:

* не потеряны ли отдельные категории рабочих
* не склеены ли два рабочих в одну строку

---

## 4. Проверка примечаний

Для `paragraph_notes`:

* там должны быть только реальные примечания и коэффициенты
* не должно быть заголовков таблиц
* `code` должен быть либо пустым, либо в согласованном виде (`ПР-1`, `ПР-2` и т.п.)
* `coefficient` должен быть только там, где это правда коэффициент
* текст примечания не должен дублировать `application_notes`

Особенно важно:

* не перепутаны ли примечания параграфа с пояснениями к таблице
* не попали ли примечания в `work_items`

---

## 5. Проверка таблиц как сущностей

Для `norm_tables`:

* у каждого параграфа таблицы реально соответствуют документу
* `table_order` идёт без скачков
* `row_count` совпадает с фактическим числом `norm_rows`
* `title` — это заголовок таблицы, а не случайный кусок текста
* если в документе есть `Продолжение табл.`, это должно быть отражено осмысленно

Особенно смотреть:

* параграфы, где больше одной таблицы
* продолжения таблиц
* таблицы с подрежимами: немерзлые / мерзлые, вручную / механизировано

---

## 6. Проверка шапок таблиц

Это один из самых важных блоков.

Для `norm_columns` проверить:

* каждая колонка реально есть в документе
* не потеряна ни одна колонка
* нет дублей вроде двух `I` при отсутствии `II`
* `column_order` идёт последовательно
* `column_key` уникален внутри таблицы
* `header` — это именно подпись колонки
* `label` не дублирует бессмысленно `header`

Особенно опасные случаи:

* группы `I / II / III / IV`
* `Iм / IIм / IIIм / IVм`
* таблицы с многоэтажной шапкой
* пустой последний столбец, который на самом деле должен быть `№` или диапазоном

---

## 7. Проверка строк таблиц

Для `norm_rows`:

* каждая запись — это реальная строка нормы, а не строка шапки
* не должно быть строк вроде:

  * `Группа грунта`
  * `При разрыхлении грунта вручную`
  * `Способ разрыхления`
* `row_order` должен идти последовательно
* `source_row_num` не должен быть случайным текстом
* строка должна иметь смысл как условие применения нормы

Проверять особенно:

* диапазоны:

  * `До 0,4`
  * `Св. 0,4 до 0,7`
* подвиды условий
* строки, разорванные в документе на 2 визуальные строки

---

## 8. Проверка значений ячеек

Для `norm_values`:

* у каждой строки есть значения по всем нужным колонкам
* значения не дублируются без причины
* не попали в ячейки куски заголовков
* `value_text` выглядит как нормальное табличное значение, а не как абзац текста
* `value_type` используется последовательно

Что особенно важно:

* нет ли смещения на один столбец вправо/влево
* не исчезла ли одна колонка в середине таблицы
* не слиплись ли два значения в одно
* не потеряны ли дробные/десятичные значения

---

## 9. Проверка связности между таблицами

Нужно проверить логику:

* число колонок в `norm_columns` соответствует тому, как заполнены `norm_values`
* у каждой `norm_row` есть значения только для своей таблицы
* нет `column_key`, которых нет в `norm_columns`
* нет `row_id`, на которые никто не ссылается
* нет таблиц без строк или строк без значений, если в документе они должны быть заполнены

Это уже “машинная” валидация целостности.

---

## 10. Проверка типографики

Отдельно пройтись по полям:

* `title`
* `unit`
* `text`
* `value_text`

И искать:

* слитные слова
* потерянные пробелы
* странные тире
* `м2` / `м3` / `м` в неправильном виде
* смешение кириллицы и латиницы
* повреждённые диапазоны

Типичные примеры:

* `нормируютсяотдельно`
* `зачисткадна`
* `Дозированиесоставляющих`

---

## 11. Приоритет ручной проверки

Не надо вычитывать весь сборник хаотично. Лучше идти по приоритету.

### Сначала проверить:

* параграфы с несколькими таблицами
* параграфы с продолжениями таблиц
* параграфы с мерзлыми/немерзлыми грунтами
* параграфы, где были уже найденные ошибки

### Затем проверить:

* крупные технические таблицы
* параграфы с длинными составами работ
* параграфы с несколькими вариантами состава звена

### В последнюю очередь:

* простые одиночные таблицы
* параграфы без сложной шапки

---

## 12. Когда можно считать Е2 финально выверенным

Е2 можно считать финальным, если выполнены все условия:

* схема полностью каноническая
* все параграфы на месте
* `work_items`, `crew_items`, `notes` не содержат мусора
* в таблицах нет строк шапки среди данных
* шапки колонок совпадают с документом
* количество строк норм совпадает с документом
* значения не съехали по столбцам
* типографические склейки устранены
* проверены все ранее проблемные параграфы
