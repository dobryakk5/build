# Импорт ENIR в БД: как сейчас загружается `Е3`

Этот документ описывает не абстрактную схему, а фактический рабочий процесс, которым сейчас загружаются параграфы `Е3` в БД.

## Главный принцип

- Эталон данных: PDF.
- Вспомогательный источник: JSON.
- В БД попадает не “как смог распарсить JSON”, а то, что подтверждено по PDF.

JSON используется как черновик:
- для быстрого поиска границ параграфа;
- для проверки названий;
- для подсказки по структуре таблиц и составу работ;
- для поиска пропущенных примечаний.

Если PDF и JSON расходятся:
- верим PDF;
- JSON либо игнорируется, либо используется только как вспомогательная заметка.

## Общая логика импорта

Импорт идёт в два слоя:

1. Структура сборника:
- `enir_collections`
- `enir_sections`
- `enir_chapters`
- `enir_technical_coefficients`

2. Данные конкретного параграфа:
- `enir_paragraphs`
- `enir_source_work_items`
- `enir_work_compositions`
- `enir_work_operations`
- `enir_source_crew_items`
- `enir_crew_members`
- `enir_source_notes`
- `enir_notes`
- `enir_norm_tables`
- `enir_norm_columns`
- `enir_norm_rows`
- `enir_norm_values`

Каждый параграф грузится отдельным SQL-файлом и может быть переимпортирован независимо:

- в начале файла делается `DELETE FROM enir_paragraphs WHERE code = 'Е3-N'`
- каскадно удаляются дочерние записи
- параграф вставляется заново в чистом виде

Это даёт повторяемый импорт без ручной чистки дочерних таблиц.

## Порядок работы по одному параграфу

### 1. Сначала читается PDF

Из PDF извлекается:
- заголовок параграфа;
- измеритель;
- указания по применению норм;
- состав работ;
- состав звена;
- таблицы норм;
- примечания;
- ссылки на ТЧ/ПР.

Практически это делается через `pdftotext -layout`, чтобы сохранить табличную сетку.

### 2. Потом сверяется JSON

JSON используется для проверки:
- не пропущен ли блок состава работ;
- не потеряна ли строка примечания;
- правильно ли понято разбиение на строки таблицы;
- нет ли альтернативной нормализованной формулировки.

Но численные значения из JSON не принимаются без сверки с PDF.

### 3. После этого собирается `seed_e3_paragraph_N.sql`

Один SQL-файл соответствует одному параграфу.

Файл почти всегда имеет одинаковую структуру:

1. `BEGIN`
2. `DELETE FROM enir_paragraphs WHERE code = ...`
3. вставка `enir_paragraphs`
4. вставка application notes / technical characteristics
5. вставка `enir_source_work_items`
6. вставка `enir_work_compositions`
7. вставка `enir_work_operations`
8. вставка `enir_source_crew_items`
9. вставка `enir_crew_members`
10. вставка `enir_source_notes`
11. вставка `enir_notes`
12. вставка `enir_norm_tables`
13. вставка `enir_norm_columns`
14. вставка `enir_norm_rows`
15. вставка `enir_norm_values`
16. `COMMIT`

## Как заполняются таблицы

### `enir_paragraphs`

Сюда пишется:
- код параграфа, например `Е3-17`
- заголовок
- измеритель
- `section_id`
- `chapter_id`
- `sort_order`

`sort_order` идёт по фактическому порядку в сборнике.

### `enir_source_work_items`

Это сырой слой.

Туда кладётся JSON-строка с:
- `condition`
- массивом `operations`

Пример:

```json
{
  "condition": "При укладке ступеней (строки № 7-12)",
  "operations": [
    "1. Установка ступеней на место с подгонкой их.",
    "2. Заделка раствором щелей между проступью и подступенком."
  ]
}
```

### `enir_work_compositions` и `enir_work_operations`

Это уже нормализованный слой того же состава работ.

- один смысловой блок условий -> одна запись в `enir_work_compositions`
- каждая операция внутри блока -> запись в `enir_work_operations`

### `enir_source_crew_items`

Сохраняется исходный состав звена почти “как в книге”:
- профессия
- разряд
- количество
- `raw_text`

### `enir_crew_members`

Это нормализованный слой состава звена.

Если в PDF в одном параграфе несколько вариантов звена, в БД всё равно сохраняются отдельные записи по профессии/разряду/количеству. Тонкие различия по области применения остаются в `raw_text` и/или `params` строк нормы.

### `enir_source_notes` и `enir_notes`

Каждое примечание сохраняется дважды:

- `enir_source_notes` как сырой источник;
- `enir_notes` как рабочий нормализованный слой.

Если примечание просто текстовое:
- `coefficient = NULL`
- важный смысл можно дополнительно положить в `conditions`

Если примечание множитель:
- `coefficient` заполняется;
- `pr_code` или `code` хранит `ПР-1`, `ПР-2` и т.д.;
- в `conditions` кладётся машиночитаемое условие применения

Если примечание не множитель, а добавка:
- `coefficient = NULL`
- в `conditions` кладутся поля `add_n_vr`, `add_rate`, `per_unit` и т.п.

Примеры:

```json
{"depth_m":{"gt":1.2}}
```

```json
{"has_openings":true,"exclude_row_nums":[3]}
```

```json
{"add_n_vr":0.11,"add_rate":0.07,"per_unit":"1m_beam","worker_grade":2}
```

### `enir_norm_tables`

Один параграф может иметь одну или несколько таблиц.

Каждая таблица получает:
- `source_table_id`
- `title`
- `sort_order`
- `row_count`

### `enir_norm_columns`

Колонки создаются вручную по смыслу PDF.

Есть два основных паттерна:

1. Табличные колонки-диапазоны:
- `thickness_380_460`
- `rect_upto_1520`
- `brick_250x120x65`

2. Логические колонки карточной строки:
- `element_name`
- `unit`
- `norm_time`
- `price_rub`

### `enir_norm_rows`

Каждая строка таблицы получает:
- стабильный `source_row_id`
- `source_row_num`
- `sort_order`
- `params JSONB`

`params` заполняются не текстом из PDF, а нормализованным машиночитаемым смыслом строки.

Примеры:

```json
{"shape":"rectangular","size_metric":"perimeter_mm"}
```

```json
{"material":"brick","thickness_bricks":0.25,"subtype":"solid"}
```

```json
{"row_range":[7,12],"step_type":"winder"}
```

### `enir_norm_values`

Тут хранится фактическое содержимое ячеек таблицы.

Используются три паттерна:

1. `value_type = 'cell'`
- когда таблица строковая или “карточная”

2. `value_type = 'n_vr'`
- когда в одной колонке хранится норма времени

3. `value_type = 'rate'`
- когда в той же колонке отдельно хранится расценка

`value_text` сохраняет оригинальный вид:
- `0-43,5`
- `1,2`

`value_numeric` хранит число для расчётов:
- `0.435`
- `1.2`

Это нужно, чтобы не парсить строку при каждом расчёте.

## Как учитываются ТЧ и ПР

### Локальные ПР

Локальные коэффициенты параграфа живут в:
- `enir_source_notes`
- `enir_notes`

Именно туда пишутся:
- множители;
- add-on примечания;
- исключения;
- диапазоны применимости.

### Техническая часть (ТЧ)

ТЧ хранится отдельно:
- `enir_technical_coefficients`
- `enir_technical_coefficient_paragraphs`

Принцип такой:

- truly global ТЧ -> запись уровня сборника
- ТЧ для списка параграфов -> одна запись коэффициента + M:N привязки в `enir_technical_coefficient_paragraphs`

Например `ТЧ-7` и `ТЧ-8` не глобальны для всего `Е3`, а привязаны только к:
- `Е3-3`
- `Е3-5`
- `Е3-6`
- `Е3-8`

Это важно: они не дублируются текстовыми массивами кодов, а связываются по FK через отдельную таблицу.

## Как принимаются решения по спорным местам

### Если PDF даёт многоуровневую таблицу

Есть два способа:

1. Разложить в одну широкую таблицу с диапазонными колонками.
2. Разложить в “карточную” форму:
- строка = одна позиция
- колонки = `element_name`, `unit`, `norm_time`, `price_rub`

Выбор зависит от того, что лучше сохраняет смысл PDF и проще ищется по `params`.

### Если в книге прочерк

Прочерк не вставляется как число.

Обычно:
- запись `enir_norm_values` для такой ячейки не создаётся
- либо строка моделируется так, чтобы отсутствующая ячейка не требовалась

### Если в книге текст неявно относится к нескольким строкам

Смысл восстанавливается вручную по PDF:
- общий заголовок переносится в `detail`
- подтип кладётся в `params`
- единица измерения дублируется на все строки, если в таблице она общая

## Как выполняется загрузка

Обычно так:

1. один раз загружается [seed_e3_structure.sql](/Users/pavellebedev/Desktop/proj/build/Enir/seed_e3_structure.sql)
2. затем по очереди запускаются `seed_e3_paragraph_*.sql`

Практически:

```sql
psql ... -v ON_ERROR_STOP=1 -f seed_e3_structure.sql
psql ... -v ON_ERROR_STOP=1 -f seed_e3_paragraph_1.sql
psql ... -v ON_ERROR_STOP=1 -f seed_e3_paragraph_2.sql
...
```

Каждый файл запускается отдельно, чтобы ошибка в одном параграфе не ломала весь пакет.

## Как делается проверка после загрузки

После каждого параграфа делается минимум три проверки.

### 1. Проверка counts

Смотрим, что действительно появились:
- `source_work`
- `work_operations`
- `source_crew`
- `source_notes`
- `norm_rows`
- `norm_values`

### 2. Проверка ключевых чисел

Делается SQL-выборка по:
- `source_table_id`
- `source_row_id`
- `source_column_key`
- `value_numeric`

И сравнивается с PDF.

### 3. Проверка коэффициентов

Смотрим:
- правильно ли сохранён `ПР-*`
- не попал ли add-on в `coefficient`
- не применяются ли `ТЧ-7/ТЧ-8` лишним параграфам

## Коротко: фактическая стратегия

- PDF определяет истину.
- JSON ускоряет разметку, но не подменяет PDF.
- Каждый параграф импортируется отдельным `seed_e3_paragraph_N.sql`.
- В БД сохраняются и сырой слой, и нормализованный слой.
- Все коэффициенты, которые можно вычислять автоматически, переводятся в `conditions JSONB`.
- Все числа сохраняются и как текст, и как numeric.
- После каждой загрузки идёт SQL-проверка по counts и по конкретным значениям.

## Что это даёт

Такой подход позволяет:
- переимпортировать любой параграф без касания остальных;
- видеть исходный текст и нормализованное представление одновременно;
- строить автоматический расчёт по `params + conditions`;
- не терять нюансы PDF, которые обычно ломаются при “слепом” импорте из JSON.
