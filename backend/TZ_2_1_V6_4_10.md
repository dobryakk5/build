# 1. Цель

Доработать импорт и классификацию строк сметы так, чтобы система независимо и корректно определяла:

1. роль строки;
2. производственную операцию;
3. объект, к которому относится операция;
4. проектный этап;
5. фактический тип работы;
6. необходимость включения строки в ГПР;
7. необходимость ручной проверки;
8. отдельную уверенность выбора этапа и типа работы.

Классификация должна учитывать одновременно:

```text
собственный текст строки
× кандидаты объектов из заголовка сметного блока
× допустимые этапы и stage_options выбранного варианта
```

Название сметного блока не должно механически назначать один подтип всем его строкам. Один блок может одновременно описывать несколько объектов и этапов — например, фундамент декоративной стены и последующую кладку самой стены. Контекст нельзя превращать в постоянный слабый бонус: для generic-строк он является сигналом выбора объекта, но конкретная операция должна выбирать совместимый объект из набора кандидатов блока.

# 2. Проверенный regression-базис

Исходный PDF содержит:

- 45 страниц;
- 1 сводную страницу;
- 21 детальный сметный блок;
- 22 позиции в сводной смете.

Из 22 сводных позиций:

- 20 подтверждаются детальными блоками;
- одна позиция представлена несколькими различимыми подвариантами и должна сопоставляться по полному тексту;
- позиция `Утилизация грунта за пределы участка` не имеет детального блока и сейчас теряется.

Контрольные суммы:

```text
Заявленный итог сводной сметы:
12 725 959,27 руб.

Сумма импортируемых детальных блоков без утилизации грунта:
12 225 959,27 руб.

Потерянная позиция:
Утилизация грунта за пределы участка
200 куб.м
500 000,00 руб.

Текущее расхождение:
500 000,00 руб.

Ожидаемое расхождение после доработки:
0,00 руб.
```

Эти значения являются обязательным интеграционным regression-тестом, а не справочной информацией.

# 3. Текущее поведение кода

## 3.1. `materials_labor_pdf_parser.py`

Текущий код:

- обнаруживает страницу `СВОДНАЯ СМЕТА`;
- сохраняет только общий `grand_total`;
- полностью пропускает создание строк со сводной страницы.

В результате строка стоимостью 500 000 руб., не имеющая детального блока, исчезает из импорта.

## 3.2. `resource_classifier.py`

Текущий каскад:

1. проценты и overhead;
2. машинные/трудовые единицы;
3. mechanism/work по regex;
4. `MODE_LABOR → work`;
5. keyword fallback.

Физические логистические операции не имеют отдельного раннего правила. Поэтому строки разгрузки внутри блока `Трудозатраты` доходят до `MODE_LABOR → work`.

Одновременно:

- `Надзор за спецтехникой` уже определяется как overhead по `_OVERHEAD_RE`;
- `Доставка спецтехники` может определяться как mechanism из-за слов `спецтехника` и `трал`;
- JSON уже содержит `logistics_markers`, но parser item type с высокой уверенностью может не позволить им переопределить `work`.

## 3.3. `upload_service.py`

При предварительной классификации используется преимущественно:

```python
row.work_name
row.section
```

Поля:

```python
raw_data["spec"]
raw_data["full_name"]
```

сохраняются парсером, но не используются последовательно во всех классификаторах.

Текущий `parent_context` одновременно выполняет две разные функции:

- контекст заголовка блока;
- контекст последней уверенно классифицированной строки.

После классификации дочерней строки он может быть заменён этой строкой, из-за чего последующие строки наследуют не объект блока, а случайный предыдущий subtype.

## 3.4. `stage_classifier.py`

Вызов выполняется с текстом:

```python
" ".join((row.section, row.work_name))
```

То есть заголовок блока и собственный текст строки заранее склеиваются и теряют разные роли.

После выбора этапа `WorkTypeClassifier` повторно выбирает подтип среди:

- `stage_options`;
- `primary_work_type`;
- `related_work_types`;
- глобального результата.

Для generic-строк это может приводить к распространению основного подтипа этапа на весь блок.

## 3.5. Уверенность

Сейчас рядом с определённым типом работы может отображаться уверенность этапа. Поэтому возможен результат:

```text
Требует ручной классификации — 100%
```

Это недопустимо.

# 4. Основные архитектурные правила

## 4.1. Этап и тип работы — разные результаты

```text
work_stage_number
```

отвечает на вопрос:

> Где операция расположена в структуре проекта?

```text
work_subtype_code
```

отвечает на вопрос:

> Что фактически выполняется в этой строке?

Один этап может содержать несколько типов работ.

Пример:

| Строка | Этап | Тип |
|---|---|---|
| Бурение под буронабивные сваи | 9.4.3 | `foundation/pile_foundation` |
| Засыпка песка под фундамент | 9.4.3 | `foundation/foundation_preparation_layers` |
| Бетонные работы с армированием | 9.4.3 | `foundation/foundation_rebar_formwork_concrete` |

## 4.2. Операция и объект определяются отдельно

Собственный текст строки определяет операцию:

```text
Укладка геотекстиля
→ operation_code = geotextile_installation
```

Заголовок блока формирует не один объект, а список кандидатов:

```text
Устройство стены для АРТ-объекта
→ decorative_wall

Устройство ленточного фундамента для установки стены из блоков
→ foundation
```

Затем конкретная строка выбирает совместимый объект:

```text
Бетонные работы с армированием
× [decorative_wall, foundation]
→ foundation
→ foundation/foundation_rebar_formwork_concrete
```

```text
Выполнение кладки из керамических блоков
× [decorative_wall, foundation]
→ decorative_wall
→ landscape/decorative_block_walls
```

Для простого блока список может состоять из одного объекта:

```text
Укладка геотекстиля
× [paving_base]
→ landscape/base_geotextile_layers
```

Итоговая модель:

```text
operation_code
× section_object_candidates
× stage_options
→ selected_object_scope_code
→ stage + subtype
```

## 4.3. Фиксированный вес контекста запрещён

Не вводить правило вида:

```text
final_score = item_score + 0.25 × context_score
```

или иное постоянное ограничение контекста 20–30%.

Причина: большинство строк этой сметы являются generic-операциями и дословно повторяются в фундаментных и ландшафтных блоках.

Контекст используется по политике доказательств:

```text
явный объект в строке
→ совместимый кандидат объекта блока
→ операция строки
→ совместимые stage_options
→ контролируемый fallback
```

Правила:

1. Явный объект строки сильнее кандидатов из заголовка.
2. Для generic-операции выбирается один совместимый объект из `section_object_candidates`.
3. Если совместимы несколько объектов и нет дополнительного сигнала — `needs_review`.
4. Контекст не может заставить все операции блока получить один subtype.
5. Конфликт строки и заголовка разрешается в пользу конкретного объекта строки.
6. Один сметный блок может содержать строки разных object scope и разных этапов.
7. При сильном конфликте строку разрешается перенести в другой этап выбранного варианта.

## 4.4. Контекст блока и предыдущая работа не смешиваются

Хранить независимо:

```text
section_title_context
section_description_context
section_object_candidates
previous_work_context
```

`section_title_context` и `section_description_context`:

- сохраняются отдельно;
- не перезаписываются дочерними строками;
- могут давать разные object candidates;
- действуют до следующего сметного блока.

`section_object_candidates`:

- содержит один или несколько объектов;
- хранит источник каждого кандидата;
- не считается финальным объектом строки;
- используется resolver для выбора `selected_object_scope_code`.

`previous_work_context`:

- обновляется только уверенной физической работой;
- действует только внутри одного `section_block_id`;
- используется для явно зависимых операций и ресурсов;
- сбрасывается при переходе к следующему блоку;
- не заменяет operation-object resolution.

Склеенный `section_parent_context` разрешается сохранить только для диагностики и полнотекстового поиска. Он не является единственным источником object scope.

## 6.1. Правила `section_object_candidates`

Каждый кандидат содержит:

```text
object_scope_code
source
matched_text
confidence_score
```

Допустимые `source`:

```text
explicit_row_object
section_title
section_description
stage_context
previous_work_context
```

Требования:

1. Дубликаты одного object scope объединяются.
2. Источники не теряются при объединении.
3. `section_title` и `section_description` могут давать разные кандидаты.
4. Финальный `selected_object_scope_code` выбирается для каждой строки отдельно.
5. Один кандидат блока не должен заранее назначаться всем строкам.

## 6.2. Поля для не-work строк

Для материалов, механизмов, логистики и overhead:

```json
{
  "row_role": "logistics",
  "gpr_included": false,
  "work_type_applicable": false,

  "work_subtype_code": null,
  "parent_work_key": "pdf-page-31-block-14:work-2",
  "parent_work_stage_number": "9.4.6",
  "parent_work_subtype_code": "landscape/decorative_block_walls"
}
```

Не копировать subtype родителя в собственное поле `work_subtype_code`. Контекст затрат хранить в отдельных `parent_*` полях.

# 7. Scope выбранного варианта

## 7.1. Правило определения category scope

Scope варианта определяется в следующем порядке:

1. Если у `project_variant` заполнено `allowed_categories`, использовать только его.
2. Если `allowed_categories` отсутствует или пуст, собрать категории из `stage_options`.
3. Категория stage option определяется так:

```text
stage_option.category
→ существующее правило _stage_option_category()
→ section_id как последний совместимый fallback
```

4. Источник scope сохранять:

```text
allowed_categories
stage_options
stage_options_inferred_section
```

## 7.2. Pair scope

Допустимые пары `section_id/subtype_id` формируются из:

- `stage_options`;
- `primary_work_type`;
- `related_work_types`.

Категорийный scope не расширяет pair scope.

Финальный кандидат допустим только если:

```text
его category входит в category scope
И
его section_id/subtype_id входит в pair scope варианта
```

## 7.3. Изменение схемы JSON

В `project_hierarchy_schema` добавить:

```json
{
  "project_variant": {
    "optional": [
      "allowed_categories"
    ]
  },
  "stage_option": {
    "optional": [
      "category"
    ]
  }
}
```

Для новой версии словаря валидатор должен:

- проверять тип `allowed_categories`;
- проверять непустые строковые значения;
- логировать `scope_source`;
- не допускать пустой pair scope;
- не выполнять молчаливый unrestricted global fallback.

Для варианта 9.4 должен получаться непустой scope даже при отсутствии явного `allowed_categories`.

# 8. Изменения `materials_labor_pdf_parser.py`

## 8.1. Полный текст строки

Для каждой строки формировать:

```python
item_text = " ".join(
    value.strip()
    for value in (name, spec)
    if value and value.strip()
)
```

Сохранять:

```python
raw_data.update(
    {
        "spec": spec or None,
        "full_name": item_text,
        "item_text": item_text,
    }
)
```

Все следующие классификаторы обязаны использовать `item_text`, а не только `work_name`.

## 8.2. Структура сметного блока

Заголовок вида:

```text
СМЕТА
Устройство стены для АРТ-объекта
Устройство ленточного фундамента для установки стены из блоков
```

разбирать на отдельные поля:

```json
{
  "section_title": "Устройство стены для АРТ-объекта",
  "section_description": "Устройство ленточного фундамента для установки стены из блоков",
  "section_parent_context": "Устройство стены для АРТ-объекта Устройство ленточного фундамента для установки стены из блоков",
  "section_object_candidates": [
    {
      "object_scope_code": "decorative_wall",
      "source": "section_title"
    },
    {
      "object_scope_code": "foundation",
      "source": "section_description"
    }
  ]
}
```

Требования:

1. `section_title` и `section_description` не объединять до object detection.
2. `section_parent_context` использовать только как вспомогательное диагностическое поле.
3. Не считать, что один блок содержит только один object scope.
4. Для обычного блока список кандидатов может содержать один элемент.
5. Материал блока может дополнять object candidate для зависимой строки, но не должен заменять заголовок.

## 8.3. `section_block_id`

## 8.3. `section_block_id`

При обнаружении нового заголовка `СМЕТА` создавать новый стабильный идентификатор блока.

Пример:

```text
pdf-page-23-block-10
```

Все строки продолжения на следующей странице должны сохранить тот же `section_block_id`.

При появлении нового заголовка:

- создать новый block id;
- сбросить `current_mode`;
- сбросить `previous_work_context`;
- заменить `section_parent_context`.

## 8.4. Парсинг сводной страницы

Убрать безусловное:

```python
if "СВОДНАЯ СМЕТА" in page_text:
    record_total()
    continue
```

Вместо этого:

1. сохранить общий `grand_total`;
2. извлечь все строки сводной таблицы в `summary_rows`;
3. не добавлять их сразу в основной список;
4. после разбора детальных блоков выполнить reconciliation.

Структура сводной строки:

```json
{
  "summary_name": "Пошаговые плиты",
  "summary_spec": "Основание пошаговых плит в газоне",
  "summary_item_text": "Пошаговые плиты Основание пошаговых плит в газоне",
  "unit": "кв.м",
  "quantity": 18.1,
  "total_price": 66247.92,
  "source_mode": "summary",
  "summary_only": true
}
```

## 8.5. Сопоставление сводной строки с детальным блоком

Основной ключ:

```text
normalized(summary_name + summary_spec)
+
точная сумма с допуском 0,01 руб.
```

Дополнительные признаки:

- единица измерения;
- количество;
- нормализованные `section_title + section_description`;
- token similarity полного текста.

Приоритет:

1. exact full text + exact amount;
2. high text similarity + exact amount + совместимая unit/quantity;
3. name + spec + amount;
4. только name — недостаточно;
5. только amount — недостаточно.

Порог fuzzy-сопоставления должен быть вынесен в константу и покрыт тестами.

## 8.6. Четыре позиции «Пошаговые плиты»

Обязательное правило:

```text
Пошаговые плиты
```

не является уникальным ключом.

Различать по `summary_spec`:

```text
Основание пошаговых плит на мягкой отмостке
Основание пошаговых плит в газоне
Основание пошаговых плит в отсыпке
Основание пошаговых плит в мульче
```

Каждая позиция должна сопоставиться со своим детальным блоком ровно один раз.

## 8.7. Summary-only строка

Если детальный блок не найден, создать обычную импортируемую строку:

```json
{
  "source_mode": "summary_only",
```

## 11.5. Выбор объекта внутри multi-object блока

Алгоритм:

```python
operation = detect_operation(item_text)
row_objects = detect_explicit_row_objects(item_text)
block_objects = section_object_candidates

objects = merge_candidates(row_objects, block_objects)

compatible = [
    obj
    for obj in objects
    if operation_object_matrix.supports(operation, obj)
]

selected_object = resolve_object(compatible)
```

Правила:

1. Если explicit row object даёт один совместимый объект — выбрать его.
2. Если операция совместима только с одним кандидатом блока — выбрать его.
3. Если совместимы несколько кандидатов с одинаковой силой — `needs_review`.
4. Если ни один кандидат не совместим — проверить stage options и controlled fallback.
5. Выбор одной строки не удаляет остальные candidates блока для следующих строк.

## 12.1. Новая сигнатура

```python
def classify_row_to_stage(

    previous_work_context: dict | None = None,
    *,

    global_result: ClassificationResult | None = None,

```

Не передавать `section + work_name` в `item_text`.

## 12.2. Выбор этапа

Этап выбирается по:

1. явному объекту строки;
2. операции;
3. совместимому object candidate блока;
4. совместимым `stage_options`;
5. scope варианта;
6. последовательному контексту только как дополнительному признаку.

Один блок не закрепляется заранее за одним этапом. Разные строки одного `section_block_id` могут получить разные этапы.

## 12.3. `grouped_all`

Для этапов с `stage_options_mode = grouped_all`:

1. parent stage не назначает `primary_work_type` всем строкам;
2. каждая work-строка разрешается до конкретного stage option;
3. `primary_work_type` используется только как reference;
4. если option не определён, subtype не записывается автоматически;
5. уверенный глобальный subtype сохраняется, если допустим;
6. generic-строка разрешается через operation-object matrix;
7. неизвестная строка получает review;
8. stage option `decorative_block_walls` должен участвовать в 9.4.6 после обновления JSON.

## 12.4. Перенос между этапами

Обязательные примеры:

```text
Перемещение вынутого грунта

```

```text

```

```text

```


- между блоками;
- вместо operation-object resolution;
- для удаления альтернативных object candidates;

# 15. Уверенность и отображение результата

## 15.1. Правила

1. `unknown` не может иметь subtype confidence 100%.
2. `needs_review=true` не может отображаться как уверенный subtype.
3. Уверенность этапа не подставляется в колонку типа работы.
4. Для не-work строк subtype confidence отсутствует.
5. Ручной выбор отображается как `manual`, а не как вычисленные 100%.
6. Raw score и confidence score хранить отдельно.

# 19. Критерии приёмки

Работа считается принятой, если одновременно выполнены все условия:

1. `spec/full_name` участвует во всех классификациях.
2. Заголовок блока и текст строки не склеиваются до классификации.
3. `section_title` и `section_description` сохраняются отдельно.
4. Введены независимые `section_object_candidates` и `previous_work_context`.
5. Один сметный блок может содержать строки разных object scope и разных этапов.
6. Выбор объекта одной строки не удаляет остальные candidates блока.
7. Generic-строки разрешаются через operation × object candidates × stage_options.
8. Нет фиксированного ограничения веса контекста 20–30%.
9. Явный объект строки имеет приоритет над заголовком.
10. Добавлены операции `block_masonry` и `compaction`.
11. `grouped_all` не распространяет primary subtype на все строки.
12. Физическая логистика исключается из ГПР.
13. `Утилизация грунта` остаётся work и получает `soil_disposal`.
14. `Надзор` остаётся overhead.
15. Материалы, механизмы, logistics и overhead не получают собственного subtype.
16. Сводные строки сопоставляются по полному тексту и сумме.
17. Четыре позиции `Пошаговые плиты` не смешиваются.
18. Summary-only строка стоимостью 500 000 руб. импортируется ровно один раз.
19. Итог импорта равен 12 725 959,27 руб.
20. Финансовое расхождение равно 0,00 руб.
21. Mixed-object блок АРТ-стены распределяет фундаментные операции и кладку по разным объектам.
22. `Выполнение кладки из керамических блоков` получает `landscape/decorative_block_walls`.
23. `Трамбование` и `Вибротрамбование` дают `compaction`, но разные subtype по объекту.
24. Бетонная отмостка и основание гранитных ступеней получают этап 9.4.4.
25. Чистовая планировка получает существующий этап 9.4.11.
26. Уверенность этапа и subtype отображается раздельно.
27. Нет `needs_review/unknown` с уверенностью subtype 100%.
28. Все зеркальные unit-тесты проходят.
29. Регрессионные тесты остальных типов смет проходят.

- переписывать всю taxonomy;
